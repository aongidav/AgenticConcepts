01 — Overview
What is an Agent?
An agent is software that perceives its environment, makes decisions, takes actions, and pursues goals — with varying degrees of autonomy and without step-by-step human instruction.
🧠
Perceive
Receives inputs: messages, data, events, tool results
💭
Reason
Decides what to do next, evaluates options, plans steps
⚡
Act
Calls tools, sends messages, updates records, triggers workflows
📦
Remember
Retains context across turns, sessions, or tasks
Key distinction

Traditional software follows fixed paths. Agents follow goals. The same agent can handle a refund request, a product question, and a booking — without separate code for each.

Why this matters for product

Agents shift the design problem. Instead of designing screens and flows, you're designing goals, constraints, personality, and the surface where the agent acts. The UX challenge moves from "what does the user tap?" to "what does the agent decide, and can users trust it?"

01b — Overview
Automation vs. Agent
The distinction isn't just technical — it's a fundamentally different design contract with users about who is in control and what the system will do next.
Automation / Workflow Agent
Goal-setting Set by builder at design time Interpreted from user intent at runtime
Path Predefined, branching Emergent, decided step by step
Handles unknown inputs Falls back / errors Attempts to handle or asks for clarification
Builder models Every case explicitly Goal + constraints, agent infers cases
User trust required Low — predictable outcome Higher — outcome varies by reasoning
Best for Repetitive, well-defined tasks Varied, nuanced, context-dependent tasks
Design implication

Most real products blend both. Use automation where outcomes must be exact and auditable (billing, GDPR requests). Use agents where user needs are varied and language is the input (support, onboarding, research).

02 — Core Concepts
Anatomy of an Agent
Every agent is built from a small set of components. How you configure them determines behaviour, capability, and risk.

The five components

📋
System Prompt
Defines the agent's identity, goals, tone, and hard constraints. This is the agent's "job spec".
🔧
Tools
Functions the agent can call: search, CRM lookup, send email, create record.
🗃️
Memory
What the agent retains — from conversation history to external knowledge stores.
⚙️
Model
The underlying LLM doing the reasoning. Capability and cost vary by choice.
🛡️
Guardrails
The rules that override model decisions — refusals, human escalation, confidence thresholds.
🎯
Trigger / Entry
What activates the agent: a message, an event, a schedule, or an external webhook.
02b — Core Concepts
Tools & Actions
Tools are what give agents real-world effect. Without tools, an agent can only produce text. With tools, it can read CRM data, send emails, create tickets, and update records.

Tool permission levels

The most important design decision around tools is not which ones to add — it's what permission level each carries. This directly maps to how much risk you're accepting.

READ — Safe WRITE — Requires confirmation EXECUTE — High risk, gate carefully
Design principle

Give agents the least privileged set of tools that still lets them complete the goal. Write permissions should require a confidence threshold or explicit user confirmation before executing.

Tool call anatomy

When a model decides to use a tool, it generates a structured call: a function name, typed parameters, and the expected return shape. The result feeds back into the model's context before it continues reasoning.

User message → Model reasoning → Tool call decision
↓
Tool executes → Result injected into context → Model continues
02c — Core Concepts
Memory & Context
Memory determines whether an agent feels continuous and aware, or stateless and forgetful. There are four distinct types — each with different UX and cost implications.
TypeWhat it isPersists across?Example
In-context The current conversation window Current session only "Earlier you said you prefer email…"
External / RAG Retrieved from a knowledge base Permanently Product docs, past tickets, CRM notes
Episodic Summary of past interactions Across sessions "Last time, you were troubleshooting X"
Procedural Fine-tuned model weights Permanently (baked in) Domain expertise, brand voice
UX design note

Users often assume agents remember everything. Design for memory transparency — make it clear what the agent knows, what it's been told, and when it's guessing. This is a trust surface, not just a technical detail.

02d — Core Concepts
Reasoning & Planning
Reasoning is what separates agents from chatbots. An agent doesn't just respond — it breaks down goals, identifies what it needs to know, sequences actions, and evaluates its own outputs.

ReAct pattern (Reason + Act)

The most common agent reasoning pattern. At each step the model produces a thought (what it's considering), an action (tool call or response), and an observation (what came back). It loops until the goal is met or it's stuck.

Goal received → Think: what do I need?
↓
Act: call tool / respond → Observe: what came back?
↓
Goal met? → yes → Respond to user
└ no → loop back to Think
Design risk

Reasoning loops can go wrong: an agent can get stuck, take too many steps, or pursue a subgoal that wasn't intended. Always set a max-step limit and a fallback to human handoff.

03 — Design Decisions
Autonomy & Guardrails
The hardest design question in agent-building isn't "what can it do?" — it's "what should it do on its own, and when should it ask?" This is the autonomy dial.

The autonomy spectrum

Fully supervised Suggest + confirm Default setting for most SMBs Act + notify Fully autonomous

Most SMB customers land in the middle — they want the agent to act, but they want to know what happened and be able to undo it. Design for this by default; power users can opt into higher autonomy.

Types of guardrails

Guardrails are not error messages — they're an intentional design layer that shapes agent behaviour:

Hard stop

The agent must never do X, regardless of user instruction. Example: never delete a contact without explicit multi-step confirmation.

Soft limit + escalate

The agent can proceed but must surface confidence level. If below threshold, hand off to human or ask for clarification.

Audit trail

The agent acts but every decision is logged with reasoning, so users can review and override after the fact.

03b — Design Decisions
Triggers & Flows
Triggers are the entry points that activate an agent. The trigger type fundamentally shapes the agent's expectations, memory state, and what a "good response" looks like.
Trigger TypeExampleAgent context
ConversationalUser sends a messageHas conversation history, responds in-thread
Event-basedContact form submittedNo history, acts on event data payload
ScheduledDaily digest at 9amNo user present, acts on stored state
Tool resultAPI returns anomalyMid-flow, continues reasoning chain
Human handoffAgent escalates to humanFull context passed, human takes over
Flow design tip

Always design the exit condition first. What does "done" look like for this trigger? Then design the steps backward. Agents without clear exit conditions tend to over-act or spin.

03c — Design Decisions
Common Agent Patterns
Most real-world agents are combinations of a small set of recurring patterns. Recognising which pattern you're building helps you anticipate the UX challenges upfront.
🔁
Router Agent
Classifies intent and hands off to specialist agents or flows. Like a smart dispatcher.
🔍
Research Agent
Retrieves, synthesises, and summarises. Reads many sources, returns structured insight.
✍️
Generation Agent
Creates content: emails, summaries, campaign copy. Often uses retrieved context to personalise.
🤝
Conversational Agent
Maintains dialogue over multiple turns. Handles clarification, follow-ups, and ambiguity.
🛠️
Execution Agent
Takes action in systems: updates CRM, sends emails, creates tasks. Highest-stakes pattern.
👥
Multi-Agent
Orchestrator delegates to specialised sub-agents. Handles complex, multi-step goals.
04 — HubSpot Context
Breeze Studio Concepts
Breeze Studio is HubSpot's platform for building agents — currently positioned around customer-facing bots, internal copilots, and automated outreach. Understanding the platform shapes what patterns are feasible.

Key platform constraints to design around

🔗
CRM-native context
Agents have first-class access to contact, deal, and company records. Design for this as the memory layer.
⚡
Workflow as trigger
Agents can be wired into existing HubSpot workflow automations — this is the hybrid automation/agent model.
🧩
Copilot surface
Agents running in the CRM sidebar — assisting reps in real time rather than acting autonomously.
🏪
SMB audience
Builders are often non-technical. Prompts, not code. Trust is earned through transparency, not power.

Design challenge for the team

The hardest problem in Breeze Studio UX is the builder-user gap: the person building the agent and the person using the agent are often different people with different goals, context, and risk tolerance. Design artifacts need to serve both.

04b — HubSpot Context
SMB Trust Signals
Research consistently shows SMBs want AI to act, but they're risk-averse. The trust design surface is often underinvested — it's the difference between adoption and abandonment.

What builds trust for SMB users

Explainability

Show what the agent is doing and why — even briefly. "Looked up 3 contacts matching this criteria" is more trusted than a silent result.

Reversibility

If the agent took an action, make it undoable. The ability to undo is more trust-building than a confirmation prompt before the action.

Scope clarity

Users need to know what the agent can and cannot do. Unexpected capability surprises feel unsafe even when the outcome is positive.

Progressive autonomy

Start supervised. Let users earn trust in the agent by reviewing its decisions before granting it broader action permissions.

From user research: SMB customers are more worried about the agent doing the wrong thing confidently than about it doing nothing. Design the failure mode to be visible and recoverable.
